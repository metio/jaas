### Carried forward

#### B1 (closed) — `evaluateWithDeadline` goroutine leak

`internal/eval/`. The synchronous go-jsonnet API has no context-cancellation entry point, so an orphaned eval keeps running until it finishes naturally. We can't fix the leak itself at this layer — that needs ctx threading inside go-jsonnet — but we bound its blast radius.

**What landed.**

- `internal/eval/semaphore.go` — CAS-loop atomic counter gating concurrent calls into `EvaluateFile` / `EvaluateAnonymousSnippet`. Configured via `eval.SetMaxConcurrentEvals(n)` from `main.go`; default is `max(GOMAXPROCS*4, 16)` (sized to bound worst-case goroutine pile-up, not to clip steady-state throughput). Zero disables the gate. The slot is released *inside the eval goroutine* (via `defer release()` in the eval closure that runs inside the spawned goroutine), not in the parent caller — so a timed-out parent doesn't free its slot until the orphan actually finishes (PG1).
- `eval.ErrEvalUnavailable` — sentinel returned when the cap is full. HTTP handler turns this into 503 with `ErrCodeEvaluationUnavailable` (wire-stable, joins the four existing codes); the reconciler emits a `Warning EvalUnavailable` event, bumps `jaas_snippet_eval_unavailable_total{ns,name}`, and `RequeueAfter`s — Ready condition stays untouched because backpressure isn't failure.
- `JsonnetHandler` now routes through `eval.EvaluateFile` (previously had a duplicate local `evaluateWithDeadline` that bypassed both the semaphore AND the leak gauge — both surfaces now share one eval flow).
- Four new Prometheus metrics: `jaas_eval_in_flight` (gauge, live slot count), `jaas_eval_max_concurrent` (gauge, configured cap), `jaas_eval_unavailable_total` (process-global counter, HTTP + operator), and `jaas_snippet_eval_unavailable_total{ns,name}` (labeled operator-side counter).
- Two new PrometheusRule alerts: `JaaSEvalSaturation` on `in_flight / max_concurrent > 0.9` (guarded on `max_concurrent > 0` so a disabled gate never fires), `JaaSEvalRejected` on `rate(unavailable_total[5m]) > 0.05/s`. Thresholds tunable under `operator.metrics.prometheusRule.thresholds.evalSaturation*` / `evalRejected*`.
- `docs/runbooks/eval-saturation.md` — paths A (runaway snippet pinning slots while its parent times out — visible as elevated `jaas_eval_outstanding_timed_out`) and B (genuine load above the cap). Remediation per path: suspend + tighten timeout/MaxStack vs. raise the cap or scale horizontally.

The existing leak gauge `jaas_eval_outstanding_timed_out` is unchanged and is now the load-bearing signal for diagnosing path A — saturation alerts fire from path B; the leak gauge separates the two.

#### #11 — Watch-namespace chart RBAC pivot (closed)

The chart's `operator.watchNamespaces` value now drives both halves of the multi-tenant story.

**Code-side.** `--watch-namespaces` / `JAAS_WATCH_NAMESPACES` → `Cache.DefaultNamespaces`. Set it and the manager's informers only observe CRs in the listed namespaces. Outside-namespace CRs never reach the reconciler.

**Chart-side.** The original single `ClusterRole`/`ClusterRoleBinding` pair is split into two role bundles:

- `clusterrole-operator-cluster` — cluster-scoped resources (CRDs, optional VWC for self-signed webhook). Bound via `clusterrolebinding-operator-cluster` always.
- `clusterrole-operator-tenants` — every namespaced resource the reconciler touches (snippets, libraries, ExternalArtifact, Flux source CRs, ServiceAccounts, token-mint subresource, events). Kept as a ClusterRole (not a Role) so the same role definition can be bound either cluster-wide or per-namespace.

The tenant binding is conditional:

- `operator.watchNamespaces` empty (default) → `clusterrolebinding-operator-tenants` cluster-binds the tenants role.
- `operator.watchNamespaces: [team-a, team-b, ...]` → one `RoleBinding` per listed namespace in `rolebinding-operator-tenants.yaml`, each referencing `clusterrole-operator-tenants`. The cluster-wide tenant binding does not render. The deployment receives `-watch-namespaces=team-a,team-b,...` so the cache and the apiserver agree on the boundary.

`helm-unittest` covers both rendering modes (default cluster-wide + scoped per-namespace). `verify.yml`'s kube-score gate gained a `watch-namespaces` variant that renders the chart with `operator.watchNamespaces={team-a,team-b}` and runs the same critical-finding gate.

#### #20 — Horizontal sharding via UID hash

**What jaas-operator has.** `internal/sharding/` (FNV-1a hash of `metadata.UID` mod shard count → shard index). Gate at the top of every `Reconcile`: if the snippet's shard doesn't match this pod's `--shard-index`, return early. Workload pivots from Deployment to StatefulSet so each pod gets a stable ordinal that derives its shard-index from `POD_NAME`. Mutually exclusive with leader election.

**Why hold:**

1. **Bench data says we don't need it.** `internal/operator/bench_test.go` clocks ~828 µs/op concurrent on 16 cores. A single replica handles ~1,200 reconciles/sec sustained. A 10,000-snippet cluster reconciling every 5 minutes generates ~33 reconciles/sec — three orders of magnitude under saturation.
2. **Cost is significant and contagious.** Sharding code is the small part (~150 LOC). The chart pivot from Deployment to StatefulSet is the big one — different rollout semantics, pod identity model, scaling story (`kubectl scale statefulset --replicas=...` doesn't rebalance shards automatically), anti-affinity rules. Operators familiar with Deployment-shaped JaaS need to relearn. The `--leader-election=true && --shard-count>1` interaction is a boot-time error, which means flag validation surface.
3. **Better alternatives exist if we ever do saturate.** Raising `MaxConcurrentReconciles` (controller-runtime knob, default 1 in jaas) gives 10-20× throughput per replica before sharding becomes the right answer.
4. **The bench shows where the real bottleneck would surface first.** Storage `Put` latency dominates per-reconcile time. If saturation hits, "switch to S3 with parallel uploads" or "raise MaxConcurrentReconciles" beats sharding.

**Cost to close.** ~400 LOC (sharding package + Reconcile gate + flag validation) + chart pivot to StatefulSet + ~10 unittests + updated docs. Roughly a day end-to-end.

**Audit verdict:** STILL_OPEN. No `internal/sharding/`, no StatefulSet template, no shard flags. Matches documented status.

**Recommendation.** Don't close until proven necessary. Items N5 + N6 (cluster-wide List per source event) are the more likely cause of throughput issues at scale than per-pod concurrency — fix those first.

### New findings — detail on the actionable ones

The five immediate findings (N1, N2, N3, N4, N7) shipped in the same round; the detail on each below covers the design choice for future reference. Remaining detail covers the next-to-fix item (N8).

#### N1 — S3Backend.Put goroutine deadlock under partial-write failure (closed)

`internal/storage/s3.go`. `minio.PutObject` can return early (network error, ctx cancellation, throttling) without fully draining the `io.Pipe`. Before the fix, the writer goroutine was blocked inside `streamTarGz`'s `pw.Write` waiting for a reader that was gone; the main path was blocked on `<-writerDone` waiting for the writer to send. Deadlock — one stuck Put per snippet would eventually pin every reconcile goroutine.

**Closed:** call `pr.CloseWithError(putErr)` between `PutObject` returning and `<-writerDone` blocking. Closing the reader makes `pw.Write` return `io.ErrClosedPipe`, the writer goroutine reaches its `writerDone` send, and the main path unblocks. Harmless on the happy path — by the time the close runs, the pipe is already drained.

#### N2 — buildEntries Output=source + sourceRef → empty tarball (closed)

`internal/operator/publisher.go`. Before the fix, the `OutputSource` branch iterated `snip.Spec.Files`, which is empty when the snippet uses `spec.sourceRef`. The fetched-and-verified upstream files lived in a local map inside `resolveSnippetSource` and were never threaded into `buildEntries`. Status flipped Ready=True/Synced with the sha256-of-empty revision; downstream Flux consumers fetched a zero-entry archive.

**Closed:** `Publisher.Publish` (and the reconciler's `tracedPublish` / `publish` helpers) now take a `sourceFiles map[string]string` parameter. The reconciler threads the resolved source map — same value for inline `spec.files` and for sourceRef-fetched content — and `buildEntries` packs from that map in source mode. `TestPublish_SourceMode_PacksResolvedSourceFiles_NotSpecFiles` pins the invariant against future regression.

#### N3 — HTTP redirect bypasses urlguard (closed)

`internal/sources/sources.go`. Before the fix, `urlguard.ValidateHTTPURL` ran against the initial URL only; Go's default `http.Client` follows up to 10 redirects without re-validating. A source CR whose `status.artifact.url` passed urlguard could 302 to `http://169.254.169.254/...` (cloud metadata) or `http://kubernetes.default.svc/...` and the follow would dial it.

**Closed:** `Fetcher.New()` installs `HTTPClient.CheckRedirect = f.checkRedirect`. Every hop re-runs through `f.validateURL()` (the same hook the initial dial uses), so a 302 to a denied host is rejected before the second dial. Cap at 10 hops matches Go's default. `TestNew_InstallsCheckRedirect_ValidatesEveryHop` exercises allow, deny, and the hop-limit guard.

#### N4 — singleflight propagates first caller's ctx cancellation (closed)

`internal/operator/token.go`. Before the fix, `singleflight.Group.Do` ran `Mint` under the first caller's `ctx`. If that caller's reconcile was cancelled before the mint returned, every waiter — with healthy contexts — received `context.Canceled` from `Mint`. Symptom: unrelated snippet reconciles in the same namespace flipped Ready=False with `context canceled` even though their tenant SA was fine.

**Closed:** the mint now runs under a 30s-bounded `context.Background()` derivation, decoupling cache lifetime from any single caller's deadline. CreateToken typically returns in well under a second, so the timeout is only a hang guard. `TestTokenCache_FirstCallerCancellationDoesNotPropagateToMinter` pins this — passing an already-cancelled ctx to `Token()` must still let `Mint` see a live ctx and return a usable token.

#### N7 — JsonnetLibrary advertises Ready/Revision printcolumns, never written (closed)

`api/v1/jsonnetlibrary_types.go`. The CRD declared `Ready` and `Revision` printcolumns but no reconciler wrote them. `kubectl get jsonnetlibrary` showed blank columns forever; operators reasonably assumed the library was unhealthy or the operator was broken.

**Closed:** the two misleading printcolumns are stripped from the type and the generated CRD; only `Age` remains. The shared `SyncStatus` field and `subresource:status` marker stay so adding a `LibraryReconciler` later is purely additive — at that point, restore the printcolumns.

#### N8 — Rate-limited snippets requeue silently (closed)

`internal/operator/reconciler.go`. Before the fix, `Limiter.Reserve()` returns `RequeueAfter` but the reconciler returned without emitting an event or bumping a metric — only a Debug log fired. Operators wondering why a snippet hadn't reconciled had no visible signal.

**Closed:** the denied `Reserve` now bumps `jaas_snippet_rate_limited_total{namespace,name}` and emits a `Warning RateLimited` event with the requeue delay in its message. The Ready condition stays as-is — backpressure isn't a failure mode, and overwriting Ready=True with a transient throttle would mask the real state. `TestEnvtest_Reconcile_RateLimiterReturnsRequeueAfter` extended to assert the event + counter at the burst-exhaustion round.

#### N11 — Publisher.Withdraw Forbidden treated as transient (closed)

`internal/operator/reconciler.go` finalizer path. A Forbidden response from the ExternalArtifact Delete (revoked RBAC, missing CRD) won't heal by retry — the cluster operator has to grant the verb or install the CRD. Before the fix, the reconciler waited out the full `MaxWithdrawWait` (1h default) before force-dropping; the workqueue burned a reconcile every backoff interval against an error that wasn't going to change.

**Closed:** `isPermanentAPIError` classification on the Withdraw return triggers immediate `forceDropFinalizer`. Orphan-tarball cost is the same as the timeout path, but the cluster unblocks at once. `TestReconcile_DeletionWithdrawForbiddenForceDropsImmediately` pins the immediate drop (clock pinned 1 second past DeletionTimestamp, well inside MaxWithdrawWait).

#### N12 — Storage HTTP handler exposes directory listings (closed)

`internal/storage/backend.go`. `http.FileServer` defaults to auto-indexing directories — any in-cluster caller that could reach the storage port could enumerate every published snippet's revision history via `GET /` or `GET /<namespace>/`.

**Closed:** `Store.HTTPHandler` now wraps the FileServer with a `.tar.gz` suffix allowlist — the only filename shape `Backend.Put` produces. Directory paths (root, namespace, snippet) and unrelated extensions all return 404. `TestStore_HTTPHandler_ServesTarballAndRejectsDirectories` exercises both the served-tarball case and four denied shapes.

#### N13 — Cross-namespace library RBAC denials leak raw error (closed)

`internal/operator/reconciler.go` `resolveLibraries`. #### N14 — Cert rotation order (closed)

`internal/webhook/selfsigned/renewer.go`. Whichever step runs first opens a brief mismatch window with the other. fsnotify-driven certwatcher reload completes in single-digit milliseconds; the apiserver Patch typically takes tens of milliseconds. Writing first leaves the full patch latency as the mismatch window — webhook switches to the new cert before the apiserver knows the new CA.

**Closed:** `renewOnce` now patches VWC first, then writes cert. The mismatch window shrinks to roughly "patch latency minus reload latency". Neither order is windowless — true zero-downtime needs a dual-CA caBundle held during rotation (read current bundle, patch to OldCA+NewCA, write new cert, wait for reload, patch to NewCA only). Tracked as N14b below for a future change. `TestRenewer_RenewOnce_PatchesVWCBeforeWritingCert` uses an `orderingVWCClient` that observes whether `tls.crt` exists on disk at the moment Patch is called — must be absent for the test to pass.

##### N14b — Dual-CA rotation (closed)

`internal/webhook/selfsigned/renewer.go`. The dual-CA path implements the full 6-step sequence: read current caBundle, generate new bundle, patch with the union OldCA+NewCA, WriteTo, sleep `GuardDelay`, patch with NewCA only. `Renewer.GuardDelay` defaults to 5s in production (tests run ~1ms to stay fast). `GetVWCCABundle` is the new helper that returns the VWC's current caBundle for the union step; `combineCABundles` concatenates two PEM payloads with a separator newline. `TestRenewer_RenewOnce_DualCARotation` pins both invariants on a recording client: the first patch lands BEFORE WriteTo and contains OldCA; the second patch lands AFTER WriteTo and does not contain OldCA. Every admission request during rotation verifies under at least one of the two trusted chains — admission failures bounded by the apiserver's own retry behavior, not by the rotation window.

#### N16 — markSynced "0 files" message (closed)

`internal/operator/reconciler.go`. `markSynced` previously used `len(snip.Spec.Files)`, which is zero for every sourceRef snippet. Now takes a `sourceFileCount int` parameter threaded from the reconciler's resolved source — same value for inline and sourceRef modes. `TestReconcile_MarkSynced_PopulatesLastSyncTimeAndFileCount` pins the message format with a 2-file inline snippet ("from 2 files").

#### N17 — status.lastSyncTime wired (closed)

`internal/operator/reconciler.go` `markSynced` now stamps `latest.Status.LastSyncTime = &metav1.Time{Time: r.now()}` on every successful publish. The shared SyncStatus type already had the field declared; only the writer was missing. Same test as N16 asserts the timestamp matches the pinned `r.Clock()`.

#### N18 — `--crd-install-restart-delay` removed (closed)

The flag, its `Config.CRDInstallRestartDelay` field, `crdWatcher.restartDelay`, the `ErrCRDInstalled` sentinel, the `helm/values.yaml: operator.crdInstallRestartDelay` value, the `-crd-install-restart-delay=` arg in `deployment-jaas.yaml`, and the README's CLI flags block entry are all removed. The dynamic Flux-CRD watch has been live since the watcher refactor — the flag was a no-op shim. Pre-release: no installed base to preserve compat for. `helm/values.schema.json` regenerated via `helm-schema -c helm -k additionalProperties`.

#### N19 — Empty-source vs entry-file-missing (closed)

`internal/operator/reconciler.go`. When the resolved file map is empty, the failure message now distinguishes two cases:

- `spec.sourceRef.path %q matched no files in the artifact; widen the path filter or remove it` — when sourceRef.path was set
- `source contains no files; check the upstream artifact` — when the upstream tarball is genuinely empty

A non-empty file map missing the entry file still gets the original "source must contain %q as the snippet entry point" message — that's the right diagnosis for that case. `TestReconcile_EmptySourceTarball_DistinguishesPathFilter` exercises the path-filter branch via a fake Fetcher returning an empty map.

#### N24 — Bundle.WriteTo half-rotated window (closed)

`internal/webhook/selfsigned/cert.go`. `Bundle.WriteTo` now writes `tls.key` first, then `tls.crt`. certwatcher's fsnotify hook fires on the cert file's write event — by the time it reloads, the matching key is guaranteed to be present. Writing cert first opened a sub-millisecond window where certwatcher could load NewCert paired with OldKey, failing every TLS handshake until the key write completed.
